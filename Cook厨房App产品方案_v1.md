## 一、产品定位与设计哲学

**一句话定位**：一个住在你手机里、能记住你冰箱、能听懂你说话的"私人厨房管家"。

**三个设计原则**（贯穿所有功能取舍）：

1. **极简主义 (Less is more)**：每个新功能要回答一个问题 —— 解决了哪个具体痛点？如果模糊，就不加。
2. **闭环优先于花哨**：菜谱→购物→采买→入库→做菜→扣库存 这条链路一定要顺，比"100种菜谱推荐算法"更重要。
3. **AI是助手，不是噱头**：LLM只在能省你时间的地方出现（语音/小票/推荐），不强行让你"和AI聊天"。

**不做的事**（明确边界）：
- ❌ 社交分享、菜谱评分、社区互动 —— 个人app，不需要
- ❌ 精确卡路里/营养元素计算 —— 增加录入负担，价值低
- ❌ 多账号、权限管理 —— 单用户场景
- ❌ 复杂的"美食日记/打卡" —— 容易半途而废

## 二、核心痛点分析（来源：你已表达 + PM调研补充）

| # | 真实痛点 | 当前怎么解决（痛苦点） | 本App 如何解决 |
|---|---------|----------------------|--------------|
| P1 | 每天问"今天吃啥" | 翻菜谱、点外卖、想半天 | 基于冰箱现有食材一键 AI 推荐 |
| P2 | 冰箱里有啥不知道 | 蹲下来翻冰箱 / 全凭记忆 | 库存可视化，分区显示（冷藏/冷冻/调料） |
| P3 | 食材过期浪费 | 食材烂了才发现 | 进库自动算保质期，临期红色高亮 |
| P4 | 去超市忘买东西 | 手写小条 / 微信备忘录 | 购物清单 + 常买快捷库 |
| P5 | 超市买了东西回家手动录库存累 | 一个个敲键盘 | 拍小票 / 语音录入，AI 自动结构化 |
| P6 | 做完菜库存懒得手动减 | 长期不更新，库存不准 | 做完一键扣减，按"用量直觉"扣 |
| P7 | 想做的菜临时记不起来步骤 | 翻收藏夹、翻聊天记录 | 菜谱本地存、做菜模式步骤化展示 |
| P8 | 一周吃饭重样 | 凭印象做菜 | 记录"最近做过"，避免一周重复 |
| P9 | 临时来客人，分量不会换 | 凭感觉加量，做多/做少 | 一键 2 人份→4 人份缩放 |
| P10 | 做菜时手脏不方便操作手机 | 屏幕一直亮、油污滑屏 | 做菜模式：大字体、自动锁屏防关、计时器 |

## 六、LLM 能力设计（DeepSeek 集成）

> API: `https://api.deepseek.com/v1/chat/completions`
> 模型选用：`deepseek-chat`（通用）+ `deepseek-vl` 或类似多模态（小票OCR），如DeepSeek当前无原生vision，则用 base64 文本描述兜底或换其他 OCR。

### 场景 1：随口说生成菜谱 ⭐⭐⭐

**触发**：首页 ➕ 按钮 → 输入"我想吃番茄牛腩"
**Prompt 模板**：
```
你是一个中餐家常菜专家。用户想吃：{用户输入}。
请输出 JSON 格式菜谱：
{
  "name": "...",
  "servings": 2,
  "cook_minutes": 数字,
  "tags": [...],
  "ingredients": [{"name":"","quantity":数字,"unit":""}],
  "steps": ["步骤1","步骤2"...]
}
要求：步骤详细到油温、火候，食材分量精确到家常单位。直接返回JSON，无其他文字。
```

### 场景 2：基于冰箱推荐今晚菜 ⭐⭐⭐

**触发**：首页"AI推荐今晚菜"卡片
**Prompt 模板**：
```
我的冰箱里有以下食材（含临期天数）：
{库存JSON}

我已收藏的菜谱：
{菜谱名+食材列表}

请按以下规则推荐 3 道菜：
1. 优先用临期（≤3天）的食材
2. 优先匹配我已收藏的菜谱（标注 [收藏]）
3. 如收藏菜谱不够用，可推荐新菜（标注 [新]）

返回 JSON 数组：
[{"name":"","reason":"为什么推荐","missing":["缺的食材"],"source":"[收藏]或[新]"}]
```

### 场景 3：购物小票识别 ⭐⭐

**输入**：用户上传超市小票照片
**思路**：
- 优先用 DeepSeek 多模态（如可用）
- 兜底方案：本地 OCR（如 PaddleOCR / 百度 OCR API）→ 文本喂给 DeepSeek 结构化

**Prompt 模板**（结构化阶段）：
```
以下是超市小票 OCR 文本：
{ocr_text}

请提取购买的食材并归类，返回 JSON：
[{"name":"番茄","quantity":2,"unit":"个","category":"蔬菜","estimated_shelf_days":7}]
忽略：日期、收银员、总价等非食材信息。
```

### 场景 4：语音批量入库 ⭐⭐

**输入**：用户语音"今天买了两斤排骨、五个西红柿、一把香菜"
**实现**：浏览器 Web Speech API 转文字 → 喂给 DeepSeek
**Prompt 模板**：
```
用户用口语描述了刚购买的食材：
"{transcript}"

请提取并结构化为 JSON：
[{"name":"排骨","quantity":1000,"unit":"克","category":"肉类","estimated_shelf_days":2}]

规则：
- "两斤" = 1000克，"半斤" = 250克
- "一把" "一撮" 等保留原单位
- 自动判断保质期天数
```

### 场景 5：智能扣减库存 ⭐⭐

**触发**：做菜任务完成
**思路**：
- 用菜谱里写的 {食材, 数量, 单位} 直接扣
- 但如果菜谱写"葱花一撮"而库存是"小葱5根"，要 AI 判断扣减量

**Prompt 模板**：
```
我做了"{菜名}"，菜谱用量是：
{recipe_ingredients}

我的库存对应食材是：
{matched_inventory}

请输出扣减计划 JSON：
[{"inventory_id":"inv_xxx","deduct_quantity":数字,"reason":"..."}]
```

## 七、技术架构与部署方案

### 7.1 技术栈

| 层 | 技术 | 理由 |
|----|------|------|
| 前端 UI | Streamlit + streamlit-option-menu | 极快开发，移动端响应式 |
| LLM | DeepSeek API（OpenAI 兼容协议） | 你已有 Key |
| 语音识别 | 浏览器 Web Speech API（前端） | 免费，无需后端 |
| 图片 OCR | DeepSeek 多模态 / 备选百度OCR | 看 DeepSeek 当前能力 |
| 存储 | JSON 文件 + GitHub Gist 同步（推荐） | 单用户多设备最轻方案 |
| 部署 | Streamlit Community Cloud（免费） | 免费、易用、支持 secrets |

### 7.2 多设备同步方案（你的需求）

**推荐方案 A：GitHub Gist 作"云数据库"**
- 把 JSON 数据存为一个 secret Gist
- 启动时拉取最新，修改后写回
- 优点：免费、用 GitHub Token 鉴权、所有设备一份数据
- 缺点：并发写入可能冲突（个人单点用，不会冲突）

**备选方案 B：Supabase 免费版**
- Postgres 数据库 + REST API
- 优点：真正的多设备实时同步
- 缺点：需注册、稍重

**备选方案 C：本地存储 + 手动导入导出**
- Streamlit Cloud 启动时存内存 + Session
- 用"导出 JSON / 导入 JSON"实现手动多设备
- 优点：零外部依赖
- 缺点：手动操作有摩擦

> 👉 我推荐 **方案 A（GitHub Gist）**：最简单、零额外注册、对个人使用足够。

### 7.3 安全 / Secrets

- DeepSeek API Key、GitHub Token 都放到 Streamlit Cloud 的 **App Secrets** 里（不进 git）
- 进入 App 时简单"密码门"保护（避免被陌生人误用你的 API 额度）

### 7.4 移动端适配

- 顶部 `st.set_page_config(layout="centered")` 让内容居中、宽度合适
- CSS 注入：
  - 字体放大（移动端默认偏小）
  - 按钮加大点击区域 ≥ 44px
  - 固定底部导航条
- 测试：用 Chrome DevTools 模拟 iPhone 视口
